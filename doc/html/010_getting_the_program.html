Windows (see more about the Windows port at the bottom of the page):
  • Windows source
  • Windows 32bit installer
  • Windows 32bit executable and Windows 32bit DLLs; they should be unpacked in the same directory
If you don't want to compile from sources, you need either the installer or the pair "executable and DLLs". (If you don't know what they are, you probably want the installer.)
The source files are the same for Linux and Windows, but the archives are different because of some bugs preventing the Qt for Windows from using several of the button images that are available in Linux. Also, the install scripts are different.
Since this is my first attempt at creating distributable binaries, I'm not very confident that they are of good quality. One known issue is that they are not signed. Another issue is that the dependecies might not be best (e.g. the package requires some version of a library while most users have a different, incompatible version.) So this is an area where I could use some help, especially in creating binaries for non-openSUSE platforms.
Ubuntu and Debian notes:
I tested the Ubuntu 9.04 i386 .deb installation and it seemed OK except for a warning about the package not being authenticated, because it's not digitally signed. I'll look into this in the near future. However, the program installs and runs OK if you ignore that warning. You can read about adding repositories on Ubuntu here. The command I used when prompted for the "APT line" was:
deb http://ftp5.gwdg.de/pub/opensuse/repositories/home:/ciobi/xUbuntu_9.04/ ./
Currently the Ubuntu and Debian repositories are on a mirror server, so it takes some time for the latest packages to propagate. This is in the hope that new versions will be found automatically by Synaptic. (The repository on the main server uses redirection, which apparently isn't supported by Synaptic.) If new versions don't show up as available, please let me know.

Requirements
The development system is openSUSE 10.3. I also compiled MP3 Diags on openSUSE 11.1 and on Ubuntu 9.04, and it seemed to work fine. I expect most Linux distributions from 2007 or newer to be able to compile and run it (perhaps after some Install.sh tweaks.)
In order to compile the project, these packages should be installed:
Generic names C++ compiler, Qt4 development, zlib development, Boost development
openSUSE 10.3, openSUSE 11.1 g++, libqt4-devel, zlib-devel, boost-devel
Ubuntu 9.04 g++, libqt4-dev, zlib1g-dev, libboost1.35-dev
Fedora 10 gcc-c++, qt-devel, zlib-devel, boost-devel
Others to be determined
Qt 4.3.1 is the one that was used for development. Something above should be OK too, but there may be some UI glitches, about which I'd like to know.
If you want to use GCC 4.4 or above, you need Boost 1.37 or later (see http://bugs.xmms2.xmms.se/view.php?id=2215.)
If you find some other required package, please let me know.
Compiling & installing
After unpacking the surce to some directory, just run Install.sh in that directory. Note that double-clicking the script in some file browser may work or not, depending on the "current directory" being set properly and on whether a terminal window gets opened so you can type the root password to get the executable copied to /usr/local/bin. Other options are to manually run the relevant 3 commands in Install.sh, or to compile the project from KDevelop 3.5, if you have it (though you might still have to change the name of qmake.)
Uninstallation can be done through Uninstall.sh. You probably created some data files, and if you are sure you no longer need them you should remove them manually. Note that the data files come in pairs. Although you only specify an ".ini" file when you start the program for the first time (or when you create a new session), two files get created. The ".ini" only stores settings, while the actual data is stored in a file in the same directory and with the same name but with the extension ".dat".
Subversion information
The source code may also be downloaded from the Subversion server provided by SourceForge, by using this command:
svn export https://mp3diags.svn.sourceforge.net/svnroot/mp3diags
Builds created from code from this server are supposed to be of the same quality as official releases. Changes are tested before checking in.
The Subversion repository is supposed to store all the tools that are needed to build the project, not just the source code. So you can find there the documentation, installer and build scripts, Windows-specific icons, and desktop integration files. The build scripts can be used as a starting point for creating binary packages for other .rpm or Debian - based distributions.
About the Windows version
It took some fiddling with Qt, GCC, and Boost, to get a Windows version working. The first executable crashed several times for no apparent reason. After changing some icon files that didn't show up properly the crashes no longer happened, but I don't feel very confident that the problem is gone, so I recommend using Linux instead, if at all possible. At any rate, I did process some 1600 songs with the Windows port to fix issues in the audio part, and then I corrected the ID3V2 tags and added images for 35 of my albums and everything went OK, no crash whatsoever. But, as they say, your mileage may vary.
One thing that bothers me is a 1-second delay between the time when the frame of a window appears and the time the content of that window is shown properly. This makes MP3 Diags appear as running on some emulator. I (sort of) got used to it, though.
I tested a previous version of the executable on XP SP1, XP SP2, XP SP3 and some Vista Home. For some reason the SP3 machine was unable to display any text, showing little rectangles instead; however, I suspect that this is related to something wrong with that particular machine, because the other ones ran OK (well, except for the random crash I talked about.) I didn't yet test the latest version on XP SP3 or on Vista, but hopefully I'll do it soon.
Another thing that should be mentioned is that the path separator is the forward slash '/', as on Unix, and not the backslash '\'. This is by design.
As for creating the executable - it was quite frustrating. Fortunately, now it's over. If you want to build it yourself, there is build.bat, but, as pointed out in README.TXT, you'll have to change some paths manually. I couldn't get the debugger to work, so I used "good old" tracing to a file - an experience I'm not keen on repeating anytime soon.
The executable that I provide uses Qt 4.5.1 from Qt SDK 2009.02, and was compiled with the MinGW that comes with it (3.4.2.) I did the Windows version more out of curiosity; I have no need to use it. As such, the only way for me to find out about Windows-specific bugs is to have somebody report them.
